# Agent 历史监测 TSDB 与节点身份稳定化设计

- 日期：2026-04-09
- 状态：待用户最终审阅
- 目标：
  1. 为 Agent 历史监测数据引入 TSDB 存储。
  2. 修复 Docker / watchtower / 服务端远程更新后 Agent `node id` 变化的问题。
  3. 支持长期查询网络连通性曲线与节点离线事件，并为后续 AI 查询提供结构化数据。

---

## 1. 背景与问题陈述

当前仓库存在 3 类相关问题：

1. Agent 节点身份默认保存在可执行文件旁的 `.cybermonitor-node-id` 中。对 Docker 部署来说，这个位于容器文件层，容器被 watchtower 或服务端 Docker helper 重建后，文件会随容器消失，导致 Agent 重新生成随机节点 ID，并被服务端识别为新节点。
2. 网络连通性历史数据当前主要依赖内存 `testHistory` 与 `test_history.json`。该设计同时承担“长期存储”、“前台展示”、“AI 趋势分析”三种职责，扩展性与维护性都较差。
3. 公共前台展示页当前没有一条可靠的“按节点、按时间范围、向服务端拉取历史数据”的链路，导致用户在切换 `1D / 1W / 1M / 1Y` 时，并不能稳定看到服务端真实历史，而是经常退化为浏览器本地缓存或最近 24 小时内的样本。

本设计旨在一次性解决上述 3 个问题，并保持对旧版 Agent、旧节点 ID 与旧启动方式的兼容。

---

## 2. 范围与非目标

### 2.1 本次范围

本次设计覆盖以下内容：

1. Agent 节点身份持久化规则调整。
2. Docker 无显式 `CM_NODE_ID`、无持久化文件时的稳定身份推导策略。
3. 服务端网络连通性历史迁移至 TSDB。
4. 节点离线事件写入 TSDB 并永久保留。
5. 删除单个节点时精确删除其 TSDB 数据。
6. 公共前台曲线图默认范围改为 `1D`。
7. 公共前台曲线图改为按节点、按时间范围向服务端真实取数。
8. AI 上下文新增离线统计摘要。

### 2.2 本次非目标

以下事项不在第一阶段范围内：

1. 不把 CPU / 内存 / 磁盘 / 网卡吞吐等全部实时采样统一迁入 TSDB。
2. 不在第一阶段引入外部独立数据库服务。
3. 不在第一阶段重写整个前台页面，仅调整历史取数链路与默认时间范围。
4. 不强制要求 Docker 用户必须挂载持久化卷才能稳定保留节点身份。

---

## 3. 设计总览

整体设计分为 3 条主线：

1. **Agent 身份稳定化**：
   - 新节点默认使用 UUID。
   - `node id` 默认持久化至用户 home 目录。
   - 保留旧路径兼容与自动迁移。
   - Docker 无显式 ID 且无持久化文件时，根据宿主机稳定指纹派生确定性 UUID。
   - 服务端 Docker helper 在重建容器时可选地回填 `CM_NODE_ID`，进一步降低后续重建风险。

2. **历史数据 TSDB 化**：
   - 网络连通性历史进入 TSDB，保留最近 366 天，采用滑动窗口淘汰。
   - 节点离线事件进入 TSDB，永久保留。
   - `state.json` 继续保存配置、资料、更新状态等非时序信息。

3. **前台与 AI 接入**：
   - 公共前台历史查询改为“按节点 + 按范围”的服务端接口。
   - 默认展开显示 `1D`。
   - AI 上下文新增离线次数、离线时长等摘要，支持查询单节点离线历史。

## 4. Section 1：Agent 身份持久化与 Docker 更新链路

### 4.1 节点 ID 生成规则

#### 4.1.1 保持旧值不变

如果现有节点已经存在以下任何形式的 ID，则必须原样保留，不强制转成 UUID：

- 旧版生成的 `node-xxxx` 风格值。
- 用户显式指定的业务节点名。
- 已写入旧路径文件的历史值。

这保证升级到新版后，不会因为 ID 格式切换而让原节点资产丢失、分组断裂或历史拆分。

#### 4.1.2 新节点使用 UUID

只有在首次生成新 ID 时，才改用 UUID。裸机 / 常规非 Docker 场景下，默认生成随机 `UUID v4`。

### 4.2 节点 ID 解析优先级

节点 ID 解析优先级如下：

1. 显式 `CM_NODE_ID`。
2. 显式 `CM_NODE_ID_FILE` 指向的文件。
3. 新默认路径 `~/.cybermonitor-node-id`。
4. 旧默认路径（可执行文件同目录的 `.cybermonitor-node-id`）。
5. Docker 无显式 ID 且无文件时，基于宿主机稳定指纹派生确定性 UUID。
6. 非 Docker 无显式 ID 且无文件时，生成随机 `UUID v4`。

### 4.3 新默认持久化路径

默认路径改为：

- `node id`：`~/.cybermonitor-node-id`
- `agent token`：`~/.cybermonitor-agent-token`

注意：Docker 中的 `~` 指向容器内当前运行用户的 home 目录，通常为 `/home/cm`，不等于宿主机用户的 home 目录。

### 4.4 旧路径兼容与迁移

如果命中旧路径中的 `.cybermonitor-node-id`：

1. 读取其内容作为当前 `node id`。
2. 不改变该值。
3. 自动迁移写入新默认路径 `~/.cybermonitor-node-id`。

旧环境变量写法（例如 `node-id`）继续兼容。

### 4.5 Docker 无显式 ID、无文件时的宿主机指纹 UUID

#### 4.5.1 触发条件

仅在以下条件同时满足时启用：

1. 当前部署模式为 Docker。
2. 没有显式 `CM_NODE_ID`。
3. 没有显式 `CM_NODE_ID_FILE`。
4. `~/.cybermonitor-node-id` 不存在。
5. 旧默认路径文件不存在。

#### 4.5.2 设计原因

在无挂卷场景下，容器文件层会随重建丢失。如果继续使用随机 UUID，则 watchtower 或服务端 helper 重建后，节点身份必然变化。因此这里需要一条“无状态重建仍稳定”的兜底策略。

#### 4.5.3 指纹来源优先级

宿主机稳定指纹的建议来源优先级如下：

1. `/host/etc/machine-id`
2. `/host/var/lib/dbus/machine-id`
3. `/host/etc/hostname`
4. 若以上均不可用，再使用少量稳定宿主机特征组合兜底

最终使用稳定命名空间算法生成确定性 UUID，例如 UUID v5 风格的稳定派生逻辑。

#### 4.5.4 行为预期

- 同一宿主机多次重建容器，得到相同节点 ID。
- 不同宿主机得到不同节点 ID。
- 如果后续用户显式指定 `CM_NODE_ID` 或提供持久化文件，则显式 / 文件值仍然优先。

### 4.6 `agent token` 文件保留原因

`~/.cybermonitor-agent-token` 继续保留，但原因不是“更新时 token 会变化”，而是：

1. 服务端已经支持每节点专属 `AgentAuthToken`。
2. 旧版 Agent 可先使用全局 bootstrap token 接入，再由服务端补发 dedicated token。
3. 新版 Agent 应将 dedicated token 持久化，以避免后续每次重启都回退 bootstrap token。
4. 这有助于维持认证连续性与旧版兼容迁移。

### 4.7 服务端 Docker helper 的增强策略

当服务端下发 Docker 更新任务并重建 Agent 容器时：

1. 必须继续复用旧容器的 Env、Mounts、Network、RestartPolicy。
2. 如果旧容器中没有显式 `CM_NODE_ID`，helper 可以主动补上 `CM_NODE_ID=<当前运行节点 ID>`。

这样经过第一次服务端更新后，即使后续再被 watchtower 重建，也能继续沿用已固化的 `CM_NODE_ID`。

## 5. Section 2：TSDB 存储设计

### 5.1 存储域划分

本次将历史数据拆分为两个存储域：

1. **网络连通性历史（Network History）**
2. **节点离线事件历史（Offline Events）**

两者都属于“历史系统”，但保留策略不同，因此逻辑上归一，物理上分离更稳。

### 5.2 TSDB 选型

采用内嵌 `Prometheus TSDB` 作为底层时序存储。业务代码通过一个薄封装层访问 TSDB，避免将底层 API 散布到 `server.go` 各处。

建议目录：

```text
internal/server/history/
  manager.go
  network_store.go
  offline_store.go
  query.go
  migrate.go
  node_delete.go
```

### 5.3 网络连通性历史

#### 5.3.1 存储内容

每条网络探测结果写入以下时间序列：

- `cm_network_test_latency_ms`
- `cm_network_test_packet_loss`
- `cm_network_test_availability`

标签建议：

- `node_id`
- `type`
- `host`
- `port`
- `name`

其中：

- `availability = 1` 表示探测成功
- `availability = 0` 表示探测失败

#### 5.3.2 保留策略

网络连通性历史仅保留最近 **366 天**。

保留逻辑必须是**滑动窗口淘汰**，而不是批量删除：

- 第 367 天到来时，只删除第 1 天数据
- 第 368 天到来时，只删除第 2 天数据
- 任意时刻都保留连续的最近 366 天窗口

前台曲线图的最长查看范围对应 `1Y = 366 天`。

### 5.4 节点离线事件历史

#### 5.4.1 存储目标

支持后续查询：

- 节点累计离线次数
- 最近 30 天离线次数
- 每次离线持续时长
- 最长 / 平均离线时长
- 最近一次离线何时发生

#### 5.4.2 存储模型

当节点从“离线状态”恢复时，写入一条离线事件样本：

- metric：`cm_node_offline_duration_seconds`
- label：`node_id`
- sample timestamp：恢复时间
- sample value：本次离线总时长（秒）

据此可以推导：

- 离线开始时间 = `sample_time - duration`
- 离线结束时间 = `sample_time`
- 离线次数 = 样本数

#### 5.4.3 保留策略

离线事件历史**永久保存**，不参与网络连通性 366 天滚动淘汰。

### 5.5 删除节点时的 TSDB 清理

当服务端移除某个 Agent / 节点时，需要同步删除其 TSDB 相关历史数据以释放空间。

要求：

1. 只删除该 `node_id` 的数据。
2. 不能误删其他节点数据。
3. 同时删除：
   - network history 中该 `node_id` 的 series
   - offline events 中该 `node_id` 的 series

删除方式应使用基于 `node_id` 精确 matcher 的删除逻辑，再执行 tombstone 清理 / compact。

### 5.6 历史查询链路

#### 5.6.1 当前问题

公共前台当前并没有可靠的“按节点、按时间范围向服务端拉历史”的链路，导致 `1D / 1W / 1M / 1Y` 常常只是在浏览器本地缓存上做裁剪，而不是显示服务端真实历史数据。

#### 5.6.2 目标设计

新增公共历史查询接口，按节点与时间窗口真实查询 TSDB，例如：

```text
GET /api/v1/public/nodes/{node_id}/history?range=1d
GET /api/v1/public/nodes/{node_id}/history?range=7d
GET /api/v1/public/nodes/{node_id}/history?range=30d
GET /api/v1/public/nodes/{node_id}/history?range=1y
```

内部可以统一换算成 `from/to` 时间范围。

#### 5.6.3 服务端返回结构

第一阶段对前端保持兼容，不直接暴露底层 TSDB 格式，而是继续返回可转换为现有 `TestHistoryEntry` 结构的结果，避免一次性重写整套图表渲染逻辑。

#### 5.6.4 下采样

当查询范围扩大到 `1M / 1Y` 时，服务端应进行下采样，控制返回点数，避免前端卡顿与 payload 过大。

### 5.7 前台默认显示与缓存策略

#### 5.7.1 默认显示范围

节点卡片展开后的网络连通性曲线图，默认固定显示 `1D`，不再根据历史跨度动态推断 `1W`。

#### 5.7.2 切换时间范围的行为

用户切换 `1D / 1W / 1M / 1Y` 时，前台必须重新请求对应时间范围的数据，不能只依赖本地缓存。

#### 5.7.3 前台缓存

本地缓存仍可保留，但只能作为“节点 + 时间范围”的热缓存，而不能作为历史真相来源。

推荐缓存键维度至少包含：

- `node_id`
- `range_key`

### 5.8 AI 接入

后续 AI 查询上下文新增离线统计摘要，支持回答：

- 某台节点一共离线多少次
- 每次离线持续多久
- 最近 30 天离线多少次
- 最长一次离线多久
- 最近一次离线发生在何时

建议摘要字段：

- `total_count`
- `last_30d_count`
- `last_offline_recovered_at`
- `longest_duration_sec`
- `avg_duration_sec`
- `recent_offline_sessions`

### 5.9 迁移策略

旧 `test_history.json` 在服务端启动时自动迁移到 network TSDB：

1. 检查 network TSDB 是否为空。
2. 若为空且存在旧 `test_history.json`，则执行导入。
3. 导入成功后，将原文件重命名为备份文件。
4. 导入失败时，不删除原文件，以便回退。

离线事件为新功能，无需从旧历史中迁移。

## 6. Section 3：验证与测试设计

### 6.1 验证目标

本次必须通过明确的自动化测试与场景回放，证明以下事项：

1. Docker / watchtower / 服务端远程更新后，节点身份不变。
2. 网络连通性历史采用最近 366 天滑动窗口，不会整批清空。
3. 离线事件永久保存，并能正确统计次数与时长。
4. 删除单个节点时，只删除该节点的 TSDB 数据。
5. 前台 `1D / 1W / 1M / 1Y` 会真实向服务端请求对应时间段的数据，而不是只能看到最近 24 小时。

### 6.2 Agent 身份与兼容性测试

#### 6.2.1 节点 ID 解析优先级

必须覆盖以下优先级测试：

1. `CM_NODE_ID` 优先于所有文件。
2. `CM_NODE_ID_FILE` 优先于默认 home 文件。
3. `~/.cybermonitor-node-id` 存在时直接复用。
4. 命中旧路径时自动迁移到新 home 路径。
5. 非 Docker 无现有值时生成随机 UUID。
6. Docker 无现有值时生成宿主机指纹确定性 UUID。

#### 6.2.2 旧版兼容

必须验证：

1. 旧版 `node-xxxx` 风格 ID 原样保留。
2. 旧环境变量写法继续兼容。
3. 旧版 bootstrap-only Agent 仍可接入并切换为 dedicated token。

#### 6.2.3 `agent token` 持久化

必须验证：

1. 读取已存在的 `~/.cybermonitor-agent-token`。
2. bootstrap token 切换 dedicated token 后会写回文件。
3. 后续重启继续使用 dedicated token。

### 6.3 Docker 更新与重建测试

#### 6.3.1 服务端下发更新

必须验证：

1. 服务端 Docker helper 重建 Agent 容器后，上报 `node id` 与更新前一致。
2. 原节点资料、分组、标签、更新状态不丢失。
3. 如果 helper 实现了回填 `CM_NODE_ID`，则验证新容器中确实带有该环境变量。

#### 6.3.2 watchtower 式重建

必须验证：

1. Docker 无显式 `CM_NODE_ID`、无挂卷时，重建后仍能得到稳定 `node id`。
2. 如果 home 目录已挂卷并存在 `~/.cybermonitor-node-id`，则优先复用文件值，而不是重新推导。

### 6.4 TSDB 历史测试

#### 6.4.1 网络连通性历史

必须验证：

1. 366 天内数据全部可查。
2. 第 367 天到来时，只淘汰第 1 天。
3. 第 400 天时，保留窗口应连续覆盖最近 366 天。

#### 6.4.2 离线事件历史

必须验证：

1. 节点离线恢复后生成 1 条离线事件。
2. 多次离线后，离线次数与每次时长统计正确。
3. 网络连通性 366 天滚动淘汰不会影响离线事件。

#### 6.4.3 删除单个节点

必须验证：

1. 删除 `node-a` 时，只删除 `node-a` 的 network history 与 offline events。
2. `node-b` 数据不受影响。
3. 只有全量清空操作才允许清空所有节点的历史。

#### 6.4.4 迁移

必须验证：

1. `test_history.json` 可正确迁移到 network TSDB。
2. 迁移失败时原文件保留。

### 6.5 前台曲线图与时间范围取数测试

这是本次的**硬验收项**。

#### 6.5.1 默认显示

必须验证：

1. 节点卡片首次展开时，默认范围固定为 `1D`。
2. 用户切换后，当前节点范围状态能记住。

#### 6.5.2 不同范围的真实取数

必须验证：

1. 切换到 `1D` 时，前台发起 `1D` 对应历史请求。
2. 切换到 `1W` 时，前台重新发起 `1W` 对应历史请求。
3. 切换到 `1M` 时，前台重新发起 `1M` 对应历史请求。
4. 切换到 `1Y` 时，前台重新发起 `1Y` 对应历史请求。

#### 6.5.3 修复“只能看到最近 24H”

必须验证：

1. 服务端有 7 天数据时，前台切到 `1W` 后能看到完整 7 天数据。
2. 服务端有 30 天数据时，前台切到 `1M` 后能看到完整 30 天数据。
3. 服务端有 366 天数据时，前台切到 `1Y` 后能看到完整 366 天数据。
4. 上述结果不能依赖浏览器已有缓存，刷新页面后仍能从服务端重新获取正确范围。

#### 6.5.4 前端缓存隔离

必须验证：

1. 同一节点 `1D` 与 `1Y` 缓存不互相污染。
2. 不同节点相同时间范围缓存不互相污染。

### 6.6 AI 离线统计测试

必须验证：

1. AI 上下文包含离线统计摘要。
2. AI 可回答“节点离线多少次、每次多久”。
3. 删除节点后，AI 不再读到该节点离线历史。

### 6.7 Manual Replay 场景

发布前至少执行以下人工回放：

1. 裸机升级：旧节点 ID 不变。
2. Docker 无显式 ID、无卷重建：节点 ID 保持一致。
3. 服务端下发 Docker 更新：仍映射到原节点。
4. 前台 `1D / 1W / 1M / 1Y`：都能看到对应真实历史窗口。

## 7. 实施原则

1. 先修身份稳定，再接 TSDB，再补前台历史查询链路。
2. 所有行为变化都必须优先兼容旧版 Agent 与旧节点 ID。
3. 所有新增历史读写都通过单独封装层进入 TSDB，不把底层 API 散布到 `server.go`。
4. 删除节点历史必须严格按 `node_id` 精确删除，避免误伤其他节点。
5. 前台时间范围按钮必须与真实服务端历史查询绑定，不能再以本地缓存裁剪代替长时间窗查询。

---

## 8. 风险与应对

### 8.1 Docker 无挂卷情况下的宿主机指纹波动

风险：某些宿主机环境无法稳定提供 `machine-id`，可能导致指纹不稳定。

应对：

1. 明确优先读取 `/host/etc/machine-id` 与 `/host/var/lib/dbus/machine-id`。
2. 兜底时记录日志，便于排查。
3. 一旦 helper 已回填 `CM_NODE_ID`，后续重建优先使用显式值。

### 8.2 TSDB 接入后查询与删除复杂度上升

风险：误删数据、查询范围不正确、下采样导致图表异常。

应对：

1. 建立精确的自动化测试矩阵。
2. 删除逻辑必须以 `node_id` 精确匹配。
3. 范围查询必须有时间窗断言测试。

### 8.3 前台仍然只展示 24 小时

风险：接口已存在，但前台状态管理仍复用旧缓存结构，导致 UI 上看起来还是短时间窗。

应对：

1. 将“不同范围是否发起真实请求”列为硬验收项。
2. 前端缓存必须按 `node_id + range_key` 隔离。
3. 端到端测试验证图表时间跨度，不只验证按钮状态。

---

## 9. 结论

本设计通过：

1. 将 Agent 身份从容器可写层剥离出来，并为 Docker 无卷场景提供稳定指纹兜底。
2. 将网络连通性历史与离线事件历史分别纳入 TSDB 管理。
3. 将前台历史展示改造成“按节点、按时间范围、从服务端真实取数”的模式。
4. 为 AI 查询补充结构化离线统计上下文。

最终可实现：

- Docker / watchtower / 服务端远程更新后节点身份稳定。
- 网络连通性曲线可查看最近 366 天滚动窗口数据。
- 节点离线次数与离线时长永久保存，并可被 AI 查询。
- 删除单个节点时，只清理其自身历史。
- 前台 `1D / 1W / 1M / 1Y` 均能真实展示对应时间范围数据，修复“只能看到最近 24H”的问题。
